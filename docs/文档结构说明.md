## 📊 文档分类说明

### 1. 需求文档 (`docs/requirements/`)

**作用**: 定义"做什么" - 产品的功能需求、非功能需求、用户故事

**内容**:
- 项目愿景和目标
- 产品需求规格
- 系统架构设计
- 竞品分析
- 完整交互流程
- API 接口文档
- UI/UX 规范

**文档清单**:
| 序号 | 文档名称 | 说明 | 适用人群 |
|------|---------|------|---------|
| 01 | `vision_and_guide.md` | 项目愿景、核心价值、使用指南 | 产品经理、技术负责人 |
| 02 | `requirements.md` | 产品需求文档 (PRD) | 产品经理、需求分析师 |
| 03 | `architecture_and_cicd.md` | 系统架构、技术选型、CI/CD | 技术负责人、架构师 |
| 04 | `competitor_analysis.md` | 市场分析、竞品对比 | 产品经理、市场分析师 |
| 05 | `skill_management_redesign.md` | 技能管理重设计 | 产品经理、开发工程师 |
| 06 | `competitor_analysis_v0.3.0.md` | v0.3.0 竞品分析 | 产品经理 |
| 07 | `complete_requirements.md` | **完整需求交互文档** ⭐ | 开发工程师、测试工程师 |
| 08 | `requirements_v0.3.md` | v0.3 需求文档 | 历史参考 |

**使用场景**:
- ✅ 需求评审会议
- ✅ 产品规划会议
- ✅ 技术方案设计
- ✅ 用户验收测试 (UAT)
- ✅ 需求变更追踪

### 2. 测试用例 (`docs/test_cases/`)

**作用**: 定义"怎么做" - 如何验证功能是否正确

**内容**:

- 后端单元测试 (Rust)
- 后端集成测试 (沙箱)
- 用户场景测试 (E2E)
- 测试用例模板

**文档清单**:
| 序号 | 文档名称 | 说明 | 适用人群 |

| - | `README.md` | 测试用例说明 | 测试工程师、QA |

| 02 | `backend_unit_cases.md` | 后端单元测试 (Rust) | 后端测试工程师 |

| 04 | `user_scenario_cases.md` | 用户场景测试 (E2E) | QA 工程师 |

**使用场景**:

## 🔄 文档工作流

### 需求阶段
```
产品需求 → 需求评审 → 完整需求文档 → 需求冻结
   ↓
docs/requirements/
```

### 设计阶段
```
系统设计 → 架构设计 → API 设计 → 设计评审
   ↓
docs/requirements/ + docs/development/
```

### 开发阶段
```
编码 → 单元测试 → 集成测试 → 代码审查
   ↓
docs/test_cases/ + docs/development/
```

### 测试阶段
```
测试计划 → 测试用例 → 执行测试 → Bug 修复
   ↓
docs/test_cases/
```

### 发布阶段
```
发布准备 → 发布文档 → 用户手册 → 版本发布
   ↓
docs/requirements/ + docs/development/
```

## 📝 文档维护规范

### 版本管理
- 文档版本与项目版本保持一致
- 重大变更时更新版本号
- 旧版本文档保留归档

### 命名规范
- **需求文档**: `01_xxx.md`, `02_xxx.md` (按重要性排序)
- **测试用例**: `01_xxx.md`, `02_xxx.md` (按测试层级排序)
- **开发文档**: `01_xxx.md`, `02_xxx.md` (按重要性排序)

### 更新流程
1. 修改文档前先创建 Issue
2. 在文档中注明修改人和日期
3. 重要变更需要评审
4. 更新后同步到主分支

### 文档质量
- ✅ 内容准确完整
- ✅ 格式统一规范
- ✅ 及时更新维护
- ✅ 中文编写
